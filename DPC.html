DPC++

La obtención de un alto rendimiento informático en las arquitecturas informáticas modernas actuales requiere un código optimizado, eficiente en el consumo de energía y escalable. La demanda de alto rendimiento continúa aumentando debido a las necesidades en inteligencia artificial, análisis de video, análisis de datos, así como en la informática tradicional de alto rendimiento (HPC).

La diversidad de la carga de trabajo moderna ha resultado en la necesidad de diversidad arquitectónica; ninguna arquitectura es la mejor para cada carga de trabajo. Se requiere una combinación de arquitecturas escalares, vectoriales, matriciales y espaciales (SVMS) implementadas en aceleradores de CPU, GPU, AI y FPGA para obtener el mejor rendimiento.

Hoy en día, la codificación de CPU y aceleradores requiere diferentes lenguajes, bibliotecas y herramientas. Existe una reutilización limitada del código de la aplicación en diferentes arquitecturas de destino.

oneAPI es una iniciativa intersectorial para un modelo de programación unificado abierto y basado en estándares que crea una experiencia de desarrollador común en todas las arquitecturas de aceleradores de cómputo. Su objetivo es ofrecer un modelo de programación eficiente y de rendimiento que elimina la necesidad de que los desarrolladores mantengan bases de código, lenguajes, herramientas y flujos de trabajo separados para cada arquitectura.

visión general

oneAPI consta de Data Parallel C ++ (DPC ++) y un conjunto básico de bibliotecas para dominios de procesamiento intensivo. Los programas DPC ++ aceleran un cálculo secuencial al distribuir el trabajo entre los elementos de procesamiento en un dispositivo. DPC ++ utiliza un modelo de programación en paralelo de datos. Los datos se distribuyen a través de un conjunto de elementos de procesamiento, y cada elemento de procesamiento opera en los datos en paralelo. Al utilizar un modelo de programación en paralelo de datos, el desarrollador puede escalar el paralelismo con el tamaño de los datos.

Un desarrollador puede usar DPC ++ para paralelizar explícitamente un algoritmo o usar una implementación paralela de una biblioteca oneAPI. oneAPI hace posible apuntar a CPU, GPU o FPGA con una sola implementación al proporcionar interfaces consistentes. Sin embargo, es posible que algunos dispositivos no implementen todas las funciones. Para lograr el mejor rendimiento, a menudo es necesario que el programador ajuste el código para la arquitectura de destino. Las bibliotecas oneAPI están escritas para lograr el mejor rendimiento en todas las arquitecturas compatibles sin necesidad de ajustes por parte del usuario final.

Como se muestra en la figura anterior, las aplicaciones que aprovechan el modelo de programación oneAPI pueden ejecutarse en múltiples plataformas de hardware de destino que van desde CPU hasta FPGA.

Datos en paralelo C ++ (DPC ++)

DPC ++ es el corazón de oneAPI. Los programas DPC ++ están escritos en ISO C ++ y utilizan el modelo de programación paralela Khronos * SYCL * para distribuir el cálculo entre los elementos de procesamiento en un dispositivo. DPC ++ amplía SYCL con funciones de rendimiento y productividad.

DPC ++ es una fuente única : el dispositivo y el código de host se pueden incluir en el mismo archivo fuente. Un compilador DPC ++ genera código tanto para el host como para el dispositivo. Cualquier compilador de C ++ puede compilar programas que solo utilicen el subconjunto de host de DPC ++.

En esta guía, es seguro asumir que cualquier cosa que mencione SYCL también es aplicable a DPC ++. Todo lo específico de DPC ++ se mencionará explícitamente.

Cómo utilizar este documento

Este documento es un manual de referencia para DPC ++. Está destinado a ser el documento que se debe consultar cuando codifica un algoritmo en DPC ++ y necesita comprender la funcionalidad detallada, el uso de una API o algún otro aspecto de DPC ++.

Si desea aprender DPC ++, la capacitación en línea de oneAPI es más apropiada. Las últimas fuentes de información de DPC ++ son la especificación oneAPI y la especificación SYCL . Las especificaciones están escritas para implementadores de elementos oneAPI y compiladores / tiempos de ejecución SYCL. Las descripciones son más precisas, pero pueden ser difíciles de entender si aún no es un experto en otros aspectos de DPC ++.

Código de muestra simple de DPC ++

La mejor forma de presentar DPC ++es a través de un ejemplo. Ya que DPC ++se basa en C ++ moderno, este ejemplo utiliza varias características que se han agregado a C ++ en los últimos años, como funciones lambda e inicialización uniforme. Incluso si los desarrolladores no están familiarizados con estas características, su semántica quedará clara en el contexto del ejemplo. Después de ganar algo de experiencia con DPC ++, estas nuevas características de C ++ se convertirán en algo natural.

La siguiente aplicación establece cada elemento de una matriz en el valor de su índice, de modo que a [0] = 0, a [1] = 1, etc.

    
        
        #include < CL/sycl.hpp>
        #include < iostream>
        
        constexpr int num=16;
        using namespace sycl;
        
        int main() {
        auto r = range{num};
        buffer< int> a{r};
        
        queue{}.submit([&](handler& h) {
            accessor out{a, h};
            h.parallel_for(r, [=](item<1> idx) {
            out[idx] = idx;
            });
        });
        
        host_accessor result{a};
        for (int i=0; i< num; ++i)
            std::cout << result[i] << "\n";
        }

Lo primero que hay que notar es que solo hay un archivo fuente: tanto el código host como el código acelerador descargado se combinan en un solo archivo fuente . La segunda cosa a tener en cuenta es que la sintaxis es C ++ estándar: no se usan palabras clave o pragmas nuevos para expresar el paralelismo. En cambio, el paralelismo se expresa a través de clases de C ++. Por ejemplo, el buffer La clase en la línea 8 representa los datos que se descargarán al dispositivo, y el cola la clase en la línea 10 representa una conexión desde el host al acelerador.

La lógica del ejemplo funciona de la siguiente manera. Las líneas 8 y 9 crean un búfer de 16 En telementos, que no tienen valor inicial. Este búfer actúa como una matriz. La línea 11 construye un cola, que es una conexión a un dispositivo acelerador. Este simple ejemplo pregunta DPC ++para elegir un dispositivo acelerador predeterminado, pero una aplicación más robusta probablemente examinaría la topología del sistema y elegiría un acelerador en particular. Una vez que se crea la cola, el ejemplo llama al enviar()función miembro para enviar trabajo al acelerador. El parámetro de este enviar()function es una función lambda, que se ejecuta inmediatamente en el host. La función lambda hace dos cosas. Primero, crea un accesoren la línea 12, que es capaz de escribir elementos en el búfer. En segundo lugar, llama al paralelo_para () función en la línea 13 para ejecutar código en el acelerador.

La llamada a paralelo_para ()toma dos parámetros. Un parámetro es una función lambda y el otro es el abarcar objeto "r"que representa el número de elementos en el búfer. DPC ++hace que esta lambda sea llamada en el acelerador una vez para cada índice en ese rango, es decir, una vez para cada elemento del búfer. La lambda simplemente asigna un valor al elemento de búfer usando el fuera descriptor de acceso que se creó en la línea 12. En este ejemplo simple, no hay dependencias entre las invocaciones de la lambda, por lo que DPC ++ es libre de ejecutarlos en paralelo de la manera que sea más eficiente para este acelerador.

Después de llamar paralelo_para (), la parte del código del host continúa ejecutándose sin esperar a que se complete el trabajo en el acelerador. Sin embargo, lo siguiente que hace el anfitrión es crear un host_accessor en la línea 18, que lee los elementos del búfer. DPC ++ sabe que este búfer está escrito por el acelerador, por lo que el host_accessor constructor (línea 18) hasta que el trabajo enviado por el paralelo_para ()Esta completo. Una vez que se completa el trabajo del acelerador, el código de host continúa más allá de la línea 18 y usa el fuera descriptor de acceso para leer valores del búfer.
